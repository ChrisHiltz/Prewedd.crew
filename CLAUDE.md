# PreWedd Crew — TLIC Team Operations Platform

## What This Is
Internal team platform for a wedding photography/videography studio (TLIC). Shooters (photographers, videographers) onboard through the app, manage their availability, receive wedding assignments, review couple-specific briefs, and prove they're prepared via a pre-wedding quiz. Admin staff manages the team roster, creates weddings, builds briefs, and assigns shooters from a master availability calendar.

## Why It Exists
The studio runs 100+ weddings/year with a mix of W2 employees and contractors. Currently, scheduling lives in spreadsheets, briefs are emailed Google Docs that people don't read, and availability is tracked by texting individuals. This app replaces all of that with one system.

## Architecture

### Plugin-Ready Design
This app is a MODULE in a future ecosystem. A separate "Pre-Wed App" (couple-facing) will share the same Supabase project, auth system, and database tables. PreWedd Crew is the "Team" door. Pre-Wed will be the "Couples" door. Same backend, different frontends.

**Do not build anything that would need to be torn out or migrated when Pre-Wed ships.** Shared tables (users, couples, weddings) are designed for both apps.

### Stack
- **Frontend**: Next.js 16 with TypeScript
- **Styling**: Tailwind CSS
- **Backend/DB/Auth/Storage**: Supabase (Postgres + Auth + Realtime + Storage)
- **Auth model**: Magic link only (email-based, no passwords, ever)
- **Hosting**: Vercel
- **Notifications**: Resend for email, Supabase Realtime for in-app

### CRITICAL: Next.js 16 — Use Docs, Not Training Data
Your training data for Next.js is outdated. Next.js 16 bundles its own docs at `node_modules/next/dist/docs/`. **Always read the relevant doc file before writing any Next.js code.** This is the source of truth for routing, data fetching, caching, server components, and all Next.js patterns.

### MCP Servers (configured in .mcp.json)
- **Supabase MCP**: Connected to the project's Supabase instance. Use for schema inspection, querying, and migration validation. Do not use for production data.
- **Next.js DevTools MCP**: Provides access to official Next.js docs, runtime diagnostics, and upgrade tools. Call the `init` tool at the start of every session to establish docs context.

### Vercel Agent Skills (installed globally)
- **vercel-react-best-practices**: 62 performance rules for React/Next.js. Follow these when writing components.
- **frontend-design**: Anthropic's official skill for production-grade UI. Use for all visual components.

### Key Supabase Details
- Row Level Security (RLS) is mandatory on every table
- Use Supabase Auth magic links — `signInWithOtp({ email })`
- Users have a `role` field: `shooter`, `admin`, `couple` (couple unused in MVP)
- All JSON fields (personality_scores, skill_scores, rates, brief_data, timeline) use JSONB columns
- Realtime subscriptions for: assignment changes, brief updates, quiz completions

## Conventions

### TypeScript
- Strict mode enabled
- Prefer interfaces over types for object shapes
- No `any` — use `unknown` and narrow
- Named exports only (no default exports except page components)

### Components
- Functional components with hooks only
- Server Components by default, `'use client'` only when needed
- Colocate component-specific types in the same file
- shadcn/ui as component foundation — install components as needed with `npx shadcn@latest add [component]`

### Supabase
- Use the Supabase MCP server to inspect schema and validate queries when possible
- Always use the server client in Server Components and Route Handlers
- Always use the browser client in Client Components
- Never expose service_role key to the client
- Generate types after any migration: `npx supabase gen types typescript --local > src/lib/types/database.ts`

### Styling
- Tailwind utility classes, no custom CSS files unless absolutely necessary
- Mobile-first (shooters use phones, admin uses desktop)
- Color palette via Tailwind config — navy (#1B3A5C), accent (#2E75B6), warm (#C9956B), green (#059669), red (#DC2626)

### File Naming
- Components: PascalCase (`WeddingCard.tsx`)
- Utilities/hooks: camelCase (`useProfile.ts`, `formatDate.ts`)
- Route files: Next.js conventions (`page.tsx`, `layout.tsx`, `loading.tsx`)

## Important Rules
- **Magic link auth only.** Never add password fields, OAuth providers, or social login. The entire UX premise is "click a link in your email, you're in."
- **Mobile-first for shooter screens.** Every shooter screen must look great on a phone. Admin screens are desktop-first.
- **RLS on every table.** No exceptions. A shooter must never see another shooter's data. Admins see everything.
- **Brief data is JSON.** The brief structure (couple profile, timeline, logistics) is stored as JSONB on the weddings table. This keeps it flexible as the brief format evolves.
- **Onboarding is the assessment.** The multi-step onboarding flow IS the personality and skill profiling. It's not a separate step — it's how a shooter creates their profile.
- **Don't build what we can integrate later.** No payment processing, no calendar API sync, no training modules, no matching algorithm. Those are Phase 2+. The MVP is: onboard → availability → assign → brief → quiz → ready.

## Database Tables (See docs/DATABASE.md for full schema)
- `users` — All users (shared with future Pre-Wed App)
- `shooter_profiles` — Personality scores, skill scores, roles, rates
- `couples` — Couple data from planning calls
- `weddings` — Each wedding event with brief data
- `assignments` — Shooter ↔ Wedding link with status tracking
- `blocked_dates` — Shooter unavailability
- `quiz_responses` — Pre-wedding quiz answers and pass/fail

## Don'ts
- Don't add a settings page — not needed for MVP
- Don't build analytics dashboards — spreadsheet exports work for now
- Don't add SMS notifications — email + in-app is sufficient
- Don't build a chat/messaging system — Slack exists
- Don't overcomplicate the quiz — 5-8 simple questions per wedding, admin-authored, not AI-generated
